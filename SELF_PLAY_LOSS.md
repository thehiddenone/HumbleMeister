# Self-play training objective

This document describes how self-play games contribute to training, the two
supported loss modes, and what changed from the previous version.

## Background — why self-play needs a separate objective

Training mixes two kinds of games:

- **Bank games**: real human games (ELO-filtered PGN). High-quality moves —
  imitating them is a sensible default objective.
- **Self-play games**: games generated by the current model against itself.
  Move quality is *not* trustworthy. A raw cross-entropy objective on these
  moves just reinforces whatever the model already does, good or bad.

So the self-play branch of the loss must be shaped by an *external*
quality signal — Stockfish — rather than treated as ground truth.

## The two loss modes

Defined in `src/humblemeister/config/_config.py`:

```python
class SelfPlayLossMode(str, Enum):
    VALUE_ONLY = "value_only"
    """Zero the policy loss on self-play games; only the value head trains on them."""
    ADVANTAGE_WEIGHTED = "advantage_weighted"
    """Policy gradient weighted by per-move Stockfish advantages (no label smoothing)."""
```

The mode is selected via `ChessTrainingConfig.self_play_loss_mode`; the
default is `ADVANTAGE_WEIGHTED`.

### `VALUE_ONLY`

Self-play games contribute **zero policy loss**. They train only the value
head (via MSE against Stockfish evals). The policy head sees only bank
games.

Rationale: safest option. Prevents the model from imitating its own
(possibly bad) moves at all. Useful as a sanity baseline or early in
training before policy signal is trustworthy.

### `ADVANTAGE_WEIGHTED` (default)

Both bank and self-play samples contribute policy loss, but with different
per-sample settings:

| Source    | Label smoothing | Per-move weight              |
|-----------|-----------------|------------------------------|
| Bank      | `0.1`           | Stockfish advantage weights  |
| Self-play | `0.0`           | Stockfish advantage weights  |

Bank games get label smoothing because the target is "this is the move a
~2200+ human played" — plausibly good, but not uniquely optimal. Smoothing
prevents overconfidence on a single legal move.

Self-play games get **no** label smoothing. The target move is only
followed when it was actually good (high advantage weight) and heavily
down-weighted when it was bad. Smoothing on top of advantage weighting
would muddy that signal.

## The per-move advantage weights

Both bank and self-play games share the same advantage-weight pipeline.
From `compute_move_weights` in `src/humblemeister/evaluation/_stockfish.py`:

```python
# Advantage of move t, from the mover's perspective:
#     adv[t] = -e[t] - e[t-1]
advs = torch.tensor(
    [-evals[t] - evals[t - 1] for t in range(1, N + 1)],
    dtype=torch.float32,
)

# z-score so advantage is game-relative rather than position-absolute
std = advs.std()
if std > 0:
    advs = (advs - advs.mean()) / (std + 1e-8)

# softmax → probability distribution; scale so mean weight = 1.0
weights = F.softmax(advs / temperature, dim=0) * N

# EOS token gets a neutral weight of 1.0
return torch.cat([weights, torch.ones(1)], dim=0)  # [N + 1]
```

Step by step:

1. Every position is evaluated by Stockfish (depth ≈ 5, 1-5 ms/pos).
2. Per-move advantage `adv[t] = -e[t] - e[t-1]` (the sign flip converts
   the post-move eval back into the mover's perspective). Good moves →
   positive advantage.
3. Z-score within the game — quality is relative to the rest of *this*
   game, not absolute centipawns.
4. Softmax with `advantage_temperature`, scaled so mean weight = 1.

So a game where every move is average gets uniform weights ≈ 1.0.
A game with one brilliancy and several blunders concentrates weight on
the brilliancy.

## How the branching is implemented

From `ChessTrainer.__policy_loss` in `src/humblemeister/trainer/_trainer.py`:

```python
def __policy_loss(
    self,
    logits: torch.Tensor,
    targets: torch.Tensor,
    move_weights: torch.Tensor,
    is_self_play: torch.Tensor,
    loss_mode: SelfPlayLossMode,
) -> torch.Tensor:
    vocab = self.__tokenizer.vocab_size
    pad = self.__tokenizer.PAD
    seq = targets.size(1)

    is_sp_flat = is_self_play.unsqueeze(1).expand(-1, seq).reshape(-1)  # [B*T]
    logits_flat = logits.view(-1, vocab)
    targets_flat = targets.view(-1)

    if loss_mode == SelfPlayLossMode.VALUE_ONLY:
        per_token = F.cross_entropy(
            logits_flat, targets_flat,
            ignore_index=pad, label_smoothing=0.1, reduction="none",
        )
        policy_mask = (~is_sp_flat).float()
    else:  # ADVANTAGE_WEIGHTED
        smooth = F.cross_entropy(
            logits_flat, targets_flat,
            ignore_index=pad, label_smoothing=0.1, reduction="none",
        )
        raw = F.cross_entropy(
            logits_flat, targets_flat,
            ignore_index=pad, label_smoothing=0.0, reduction="none",
        )
        per_token = torch.where(is_sp_flat, raw, smooth)
        policy_mask = torch.ones(targets_flat.size(0), device=targets_flat.device)

    pad_mask = (targets_flat != pad).float()
    denom = pad_mask.sum().clamp(min=1)
    return (per_token * move_weights.view(-1) * pad_mask * policy_mask).sum() / denom
```

Key mechanics:

- `is_self_play` is a per-game bool carried on the batch from the dataset.
  It's broadcast to per-token with `unsqueeze(1).expand(-1, seq)` so every
  token inherits its game's flag.
- In `ADVANTAGE_WEIGHTED`, we compute *both* smoothed and unsmoothed
  cross-entropy and select per token with `torch.where(is_sp_flat, raw,
  smooth)`. Slightly wasteful (both kernels run on every token) but
  trivial compared to the forward pass, and keeps the code branch-free on
  the GPU.
- `policy_mask` turns off self-play's contribution entirely in
  `VALUE_ONLY` by multiplying those rows by zero. In `ADVANTAGE_WEIGHTED`
  it's all ones.
- `move_weights` multiplies every token's loss regardless of mode. For
  self-play this is the advantage-weight signal doing all the work; for
  bank games it's a per-move importance on top of label smoothing.
- The denominator is `pad_mask.sum()` — averaged over real tokens across
  the whole batch, including masked-out self-play tokens in `VALUE_ONLY`.
  This means `VALUE_ONLY` produces a slightly smaller policy-loss
  magnitude (fewer contributing tokens, same divisor), but the gradient
  for each bank token is identical to what it would be on a bank-only
  batch.

## Value head — unchanged, trains on everything

Regardless of `self_play_loss_mode`, the value head trains on all games
that have Stockfish evals. Extracted from the main training step in
`_trainer.py`:

```python
token_mask = attention_mask.bool() & has_evals.unsqueeze(1)
if token_mask.any():
    value_loss_t = F.mse_loss(value_pred[token_mask], value_targets[token_mask])
```

Self-play games always have evals (they're computed as part of the
advantage-weight pipeline), so they always contribute to value loss.
Bank games contribute iff they went through the evaluator.

## What changed from the previous version

Previously, self-play games shared the bank-game objective (same
label smoothing, same cross-entropy). The only "protection" was the
advantage weight — good moves pulled the loss, bad moves pushed back.

Problems with that setup:

1. Label smoothing `0.1` on a self-play target means "the model's own
   move is ~90% right." On a blunder, that's actively wrong. Advantage
   weight down-weighted the token but didn't remove the smoothed-label
   pressure toward a bad move.
2. There was no switch to disable self-play policy contribution entirely,
   so you couldn't easily diagnose "is self-play policy loss helping or
   hurting?" by running a `VALUE_ONLY` comparison.

The new objective addresses both:

- `label_smoothing=0.0` for self-play in `ADVANTAGE_WEIGHTED` — the only
  thing pulling toward a self-play move is the advantage weight. A
  near-zero weight → near-zero gradient, no residual smoothing bias.
- `VALUE_ONLY` mode exists as a clean toggle to isolate the value head's
  learning from self-play policy influence entirely.

## How to pick a mode

- **Early training** (policy still weak, value signal being bootstrapped):
  prefer `VALUE_ONLY`. Self-play moves aren't trustworthy enough to learn
  policy from, but the value head can still learn "what position
  evaluations look like" from them.
- **Mid-to-late training** (policy is competent, self-play games contain
  real insight): switch to `ADVANTAGE_WEIGHTED`. The model's own strong
  moves reinforce themselves, and its weak moves are suppressed by the
  advantage weight.
- **Debugging**: run both modes on the same checkpoint for a few epochs,
  compare eval-set perplexity, and see whether self-play policy gradient
  is net-positive.

Related knobs in `ChessTrainingConfig`: `self_play_start_epoch`,
`self_play_ramp_epochs`, `self_play_max_ratio`, `advantage_temperature`,
`value_loss_weight`.
